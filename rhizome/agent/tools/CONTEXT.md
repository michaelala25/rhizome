# rhizome/agent/tools/

Tool definitions and infrastructure for the rhizome agent.

## Architecture

Tools are built via factory functions that close over `session_factory` and an optional `chat_pane`. Each tool creates its own `AsyncSession`, eliminating the need for a shared session lock. Tool visibility (`ToolVisibility`) controls which tools appear in the TUI status bar at different verbosity levels.

Each tool group has its own builder function (e.g. `build_core_tools`, `build_app_tools`). `AgentSession.__init__` assembles the full tool list by calling each builder directly.

## Modules

- **\_\_init\_\_.py** — Re-exports `ToolVisibility`, `TOOL_VISIBILITY`, and `tool_visibility` from the visibility module.
- **visibility.py** — `ToolVisibility` enum (LOW, DEFAULT, HIGH), `TOOL_VISIBILITY` registry dict, and `@tool_visibility` decorator for registering a tool's visibility level. Anthropic server-side tools (`web_search`, `web_fetch`) are pre-registered here.
- **core.py** — `build_core_tools(session_factory)` — core knowledge-base tools: `list_topics`, `list_knowledge_entries`, `create_topics` (recursive subtree creation via `TopicNode` schema), `delete_topics`, `read_knowledge_entries`, `list_flashcards` (by topic_ids or knowledge_entry_ids), `read_flashcards`. Defines `TopicNode` Pydantic model for recursive topic creation.
- **app.py** — `build_app_tools(session_factory, chat_pane)` — app control tools: `update_app_state` (consolidated topic selection, tab rename, and verbosity hint), `set_mode` (returns `Command` to update graph state), `ask_user_input` (interrupt-based multiple choice). Also defines the `Question` Pydantic model for `ask_user_input`.
- **sql.py** — `build_sql_tools(session_factory)` — last-resort SQL tool: `execute_sql` (unified query/modification tool; defaults to read-only, set `read_only=False` for INSERT/UPDATE/DELETE with preview and user confirmation via `sql_confirmation` interrupt; read queries capped at 200 rows). Schema introspection is handled by the `database_schema` guide instead.
- **flashcard_proposal.py** — `build_flashcard_proposal_tools(session_factory, answerer, comparator)` — mode-independent flashcard proposal workflow: `flashcard_proposal_create` (stages in state, with optional `validate=True` for inline clarity testing via answerer/comparator subagents), `flashcard_proposal_present` (shows via interrupt, builds diff summary of user edits), `flashcard_proposal_edit` (targeted edits/additions/deletions by stable ID), `flashcard_proposal_accept` (writes to DB). Defines `FlashcardProposalItem` TypedDict, `FlashcardInput` and `FlashcardEdit` Pydantic schemas. Available in both learn and review modes.
- **review.py** — `build_review_tools(session_factory)` — consolidated review session tools: `review_get_past_sessions`, `review_show_session_state`, `review_update_session_state` (lazily initializes session; updates scope, config, flashcard queue, and plan ad-hoc), `review_record_interaction` (conversational review only), `review_present_flashcards` (interrupt-based flashcard presentation for both critique-during and critique-after modes; the widget handles both manual and auto scoring inline and emits real 1-4 ratings, so this tool only sees scored cards — every card runs through `apply_rating` alongside `add_review_interaction` using `fsrs.Rating(score)` directly; Again ratings still re-queue at the tool level and do not touch FSRS state, matching pre-heap-rewrite semantics), `review_finish_session` (computes stats, builds and persists summary, clears state). Defines `ReviewConfigUpdate`, `ReviewFlashcardUpdate` Pydantic schemas. Phase progression is not tracked — guided by the system prompt.
- **resources.py** — `build_resource_tools(session_factory, resource_manager=None)` — resource retrieval: currently only exposes `query_resources` (top-k semantic search against `ResourceManager.vector_store`, hydrates chunk text at query time by joining chunks with `ResourceContent.raw_text`). Closes over the live `ResourceManager` so the index reference tracks UI-driven loader toggles; without a manager the tool returns an explanatory error. `add_resource`, `list_resources`, and `get_resource_info` are present in the file but intentionally commented out — `add_resource` predates the UI ingestion flow, and the other two are pure DB reads that don't reflect loader state.
