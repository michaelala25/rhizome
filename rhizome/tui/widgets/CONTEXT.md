# rhizome/tui/widgets/

Reusable Textual widgets shared across screens.

## Patterns

- **MVVM via `ViewModelBase`** (`view_model_base.py`): widgets that own non-trivial state are split into a view-model (state machine, no rendering) and a view (Textual widget, no session state). View-models subclass `ViewModelBase`, which provides two standard `CallbackGroup` observers (`dirty`, `focus`), an `emit(group, *args, **kwargs)` direct-fire method, an `emit_once(*groups)` context manager that returns an `Emitter` to coalesce emits within a synchronous chain of execution, and `subscribe` / `unsubscribe` helpers for view-side wiring. Subclasses define their own `Callbacks` enum for VM-specific groups (e.g. `CommitProposalViewModel.Callbacks.TOPIC_SELECTION_REQUESTS`). **Emitter threading rule:** helper methods that may participate in a caller's batch take an `emitter: Emitter | None = None` parameter, default `None` meaning "fire directly via `self`"; this matches the precedent in `ViewModelBase.request_focus`. **Async boundary rule:** emitters batch only a single *synchronous* chain of execution and never cross task spawns — `asyncio.create_task(...)` is the stopping point, and the spawned coroutine starts a fresh emit context (it opens its own `emit_once` if it has multiple emits to coalesce). The same applies to async callbacks fired from timers (e.g. `Flashcard._on_due_reveal`): they emit directly via `self.emit(...)`, never via a captured caller emitter. **Focus delegation:** `request_focus(emitter)` lets a parent VM ask a child VM to take focus (subscribers are typically `Widget.focus()` callbacks on the view side); `notify_focused()` is the view-side notification path so a parent VM can reconcile focus when its view receives focus first. Currently used by `FlashcardReviewViewModel`.

## Files

- **view_model_base.py** — `ViewModelBase`, `CallbackGroup`, `Emitter`. Base class for MVVM view-models in this directory; see the **Patterns** section above for the full convention.
- **query_backed_view_model.py** — `QueryBackedViewModel(ViewModelBase)`: abstract base for VMs whose state is the projection of a re-runnable async query. Owns the debounce + fetch-id staleness machinery (`_request_fetch`, `_debounced_fetch`, `_still_current`); subclasses implement `_fetch()` (stateless query) and `_process_fetched_data(result)` (apply to state). Cancellation policy: cancellable during the debounce window, orphan-and-supersede during the query phase (SQLAlchemy/aiosqlite cancellation is unsafe mid-query). Used by `browser/tab_base.py` and the linked-flashcards panel sub-VM. When a new VM needs the same "snapshot inputs, run query, apply result" rhythm, inherit from this rather than rolling fetch/staleness handling from scratch.
- **search_input/** — Shared `SearchInput(Input, Generic[VM])` widget plus the `SearchableViewModelMixin(ViewModelBase)` it's bound on. Concrete VMs that want the standard "single-line input, `enter` submits, `esc` × 2 clears with armed-clear state machine" UX mix in `SearchableViewModelMixin` (at the leaf — never on an intermediate base) and implement `set_search(query)`. Used by `browser/knowledge_entry_tab/view.py` (entries-side search) and `browser/knowledge_entry_tab/linked_flashcards/view.py` (linked-flashcards search). See `search_input/CONTEXT.md` for the full contract.
- **chat_pane.py** — `ChatPane(Widget)`: core chat UI containing the message area, chat input, and command palette. Holds per-session state (`messages`, `_agent_busy`, `_agent_worker`, `session_mode`, `active_topic`, `_agent_session`) and all message/command/topic-tree handlers. Each `ChatPane` instance represents an independent chat session with its own `AgentSession` and `CommandRegistry`; multiple instances live inside `TabbedContent` tabs in `MainScreen`. Delegates to `AgentSession.stream()` (callback-driven) for agent invocation, passing `AgentMessageHarness` callback methods directly. Handles `InterruptPending`/`InterruptResolved` messages from the harness to disable/re-enable the chat input during interrupt waits. Mirrors user/system messages into the session via `append_message()`. Command handlers live as `_cmd_*` methods (e.g. `_cmd_options`, `_cmd_rename`, `_cmd_clear`); `_set_mode()` handles mode switching and is called by both slash commands and agent tools. `_register_commands()` wires all handlers into the click-based `CommandRegistry` with proper option/argument schemas. `_handle_command()` dispatches through the registry, showing help/error text when returned.
- **status_bar.py** — `StatusBar(Static)`: persistent bar at the bottom of the chat screen with reactive `mode`, `topic_path`, `token_usage`, and `model_name` properties. Three-line layout: line 1 shows topic path (left) and model name (right), line 2 shows mode with shift+tab hint (left) and token usage with overhead breakdown (right), line 3 shows cache read/create stats (right-aligned).
- **message.py** — Chat message widget hierarchy. `ChatMessage(Widget)` is the abstract base class handling role prefixes, CSS styling, and collapse logic. Two concrete subclasses: `MarkdownChatMessage` renders content via Textual's `Markdown` widget (used for user/agent/system messages), and `RichChatMessage` renders content via `Static(Text.from_ansi(...))` for proper display of ANSI escape codes (used for command output like help text). User messages have a dark gray background with side margins; agent messages gain a thin rounded border in learn/review mode. Agent messages longer than `COLLAPSE_LINE_THRESHOLD` (4) lines show a collapse/expand button.
- **thinking.py** — `ThinkingIndicator(Static)`: animated braille-spinner shown while awaiting the agent's first token.
- **entry_list.py** — `EntryList(Widget)`: read-only, focusable widget for browsing a list of `KnowledgeEntry` objects. Displays a numbered entry list with cursor navigation and a bordered detail panel (title, meta, content). `set_entries()` replaces the displayed entries. Supports a `--compact` CSS class that hides the detail panel and numbering, rendering minimal `► title` lines with a metadata hint below (type, created date). Posts `Dismissed()` on Escape and `CursorChanged(entry)` when the cursor moves. Also exports shared color constants (`ENTRY_DIM`, `ENTRY_HINT`, `ENTRY_ACCENT`) used by both this widget and `CommitProposal`.
- **flashcard_list.py** — `FlashcardList(Widget)`: read-only, focusable widget for browsing a list of `Flashcard` objects. Similar structure to `EntryList`: numbered list with cursor navigation and a detail panel showing question, answer, testing notes, and linked entry count. Ephemeral flashcards display an `(ephemeral)` suffix in dim color. `set_flashcards()` replaces the displayed flashcards. Posts `Dismissed()` on Escape.
- **flashcard_review/view.py** — `FlashcardReview(InterruptWidgetBase)`: thin Textual view over `FlashcardReviewViewModel` (below). Subscribes a single `_refresh` listener to the VM's `dirty` observer list; `_refresh` dispatches by `vm.state` (START / REVIEWING / DONE) to a corresponding `_refresh_*` method that toggles child visibility and rewrites Static contents to mirror VM state. Live tickers for the think-time timer (100ms when `card.timer_visible`), AWAITING_REVEAL due-countdown (1s), and shared throbber (150ms — shown in counter when FRONT-without-visible-timer, and in the docked batch indicator when auto-scoring) are view-owned `set_interval` handles reconciled against VM state at the end of each refresh. Additionally subscribes `_maybe_resolve` to `dirty`: fires once when `vm.state` first reaches DONE and resolves the future with `_build_result()` whether or not the user cancelled — partial state is always returned. `_build_result` includes `completed` (False on cancel), and per-card `score` (None for non-rated outcomes), `score_label` ("easy"/"good"/"hard"/"again"/"skipped"/"auto"/None for untouched), `user_answer`, `duration`, and `fsrs_card` (final in-memory FSRS state — caller decides whether to commit). `from_interrupt(value, context)` pulls `cards` + `auto_score_enabled` from the payload and `scorer_subagent` + `session_factory` off `AgentContext`. Ctrl+c is handled by the VM's `on_key` path; `action_cancel_interrupt` is deliberately a no-op to avoid double-firing with the bubble-up event (TODO in source: remove ctrl+c binding from `InterruptWidgetBase` and rename `.cancel()`). `_AnswerInput(TextArea)` subclass intercepts Enter (posts `Submitted`, view forwards to VM as a synthetic Enter) and forwards any `alt+<letter>` keys up to the parent via `event.prevent_default()` without typing.
- **flashcard_review/view_model.py** — Pure state machine behind `FlashcardReview` (no rendering). `Flashcard` (State: FRONT / REVEALED_NOT_SCORED / REVEALED_PENDING_AUTO_SCORE / SCORED / AWAITING_REVEAL; Score: AGAIN/HARD/GOOD/EASY mapped onto FSRS ratings, plus AUTO and SKIPPED) encapsulates per-card state, owns a `Timer` (think-time) and optional due-timer, and holds two in-memory `fsrs.Card` snapshots (`_initial_fsrs_card` immutable + `_current_fsrs_card` advanced via the VM-owned scheduler on EASY/GOOD/HARD/AGAIN transitions). No DB I/O happens during the session — `reset()` rolls `_current` back to a copy of `_initial`, and the VM's optional `commit()` coroutine is the sole sanctioned write site (called by the widget's caller, never internally). The unified `_apply_rating(rating, score)` helper runs the rating through the scheduler then routes by the *resulting* FSRS state: `State.Review` → terminal SCORED; `State.Learning`/`State.Relearning` → AWAITING_REVEAL with a `_wait_until_due` task scheduled from the FSRS-returned due delta. `_wait_until_due` auto-reveals via `reveal_front` and fires the VM-wired `_on_due_reveal` callback so observers get notified of the async transition. AGAIN always lands in AWAITING_REVEAL (Again never produces Review); HARD/GOOD on a card still inside the Learning step ladder also lands in AWAITING_REVEAL. `FlashcardReviewViewModel` tracks top-level state (START / REVIEWING / DONE), `_cards` list (mutated in place by the requeue's remove+append — no separate display-order array), `_current_card_index`, and two round-buckets (`_remaining_before_batched_autoscore`, `_next_remaining_before_batched_autoscore`) of card ids. `_goto_next_unscored_card` lands the cursor on the next card still needing attention, preferring `_remaining` and falling back to `_next_remaining` so the cursor doesn't stall on a scored card when the only unscored cards are requeued and awaiting reveal. Batch auto-scoring happens when `_remaining` drains with AUTO-deferred cards present; the scorer subagent is invoked in a single call using stable card ids, results distribute through `card.set_score(...)` and the post-rating FSRS state determines whether the card lands in SCORED or AWAITING_REVEAL (cards in the latter are accumulated into `requeued_cards` and re-emplaced at the back of `_cards`). Invalid/missing scorer results are reverted to REVEALED_NOT_SCORED with `_auto_scoring_failed` latched (manual fallback). Exposes a single `dirty: list[Callable]` observer list — emitted at the end of each public transition method. `on_key(event)` dispatches by state to `_on_key_start/_reviewing/_done` handlers; key map is enter (reveal/rate), 1-4 (rate), alt+left/right (nav), ctrl+c (cancel), ctrl+k (toggle timer visibility), alt+x (reset current card, clears draft answer), alt+s (toggle skip). Convenience properties `num_remaining` and `remaining_position` support the "m/N (k/K remaining)" counter rendering on the view side.
- **topic_tree.py** — `TopicTree(Tree[Topic])`: the core interactive tree widget for browsing topics. Loads root topics on mount, lazily loads children on expand. Enter is suppressed (topic selection is handled by Ctrl+J/Ctrl+Enter in the parent container). `invalidate_and_refresh()` clears the tree and reloads from DB, preserving the cursor topic when possible. Has an `active_topic_id` reactive that colors the active topic (and ancestor icons) in azure via a `render_label` override.
- **messages.py** — Shared widget messages. `ActiveTopicChanged(Message)`: posted by `ExplorerViewer` and `ResourceViewer` when the user selects or clears the active topic; handled by a single `on_active_topic_changed` handler in `ChatPane`.
- **resource_view_model.py** — View model classes for the ResourceViewer widget family (MVVM pattern). `ResourceViewerViewModel` holds all persistent state (view mode, active topic, caches, sub-view-models) and is created by `ChatPane` to outlive the ResourceViewer widget across destroy/recreate cycles (e.g. dock position changes). Sub-view-models: `ResourceListViewModel` (resources, cursor), `ResourceLinkerViewModel` (resources, linked_ids, cursor), `ResourceLoaderViewModel` (resources, load states). Also defines `LoadState` enum (previously in resource_loader.py) to avoid circular imports.
- **resource_list.py** — `ResourceList(Widget)`: read-only, focusable widget for browsing `Resource` objects. Accepts a `ResourceListViewModel` to read/write persistent state (resources, cursor, show_ids). Displays a numbered resource list with cursor navigation and a bordered detail panel. Posts `Dismissed()` on Escape and `CursorChanged(resource)` when the cursor moves.
- **resource_linker.py** — `ResourceLinker(Widget)`: focusable checkbox list for linking/unlinking resources to a topic. Accepts a `ResourceLinkerViewModel` for persistent state. Shows all resources with `[x]`/`[ ]` checkboxes indicating link status. Space/Enter toggles the link, posting `LinkToggled(resource, linked)`. A hint line shows `N/M linked`. `set_resources(resources, linked_ids)` populates the list; `update_linked_ids(linked_ids)` refreshes checkbox state without replacing items.
- **resource_viewer.py** — `ResourceViewer(Vertical)`: docked bottom panel for browsing and linking resources to topics. Accepts a `ResourceViewerViewModel` for all persistent state (caches, view mode, active topic); scalar state (active topic, current topic ID, all resources) accessed via properties that read/write through to the VM. Three view modes cycled by Tab: topic resources (25/75 tree+resource list), link resources (25/75 tree+checkbox linker), load resources (full-width loader). Docked between chat input and status bar in `ChatPane`'s grid layout; toggled via `/resources` command or Ctrl+R. Handles `LinkToggled` events to call `link_resource_to_topic`/`unlink_resource_from_topic` with commit. Posts `Dismissed()` on Escape.
- **explorer_viewer.py** — `ExplorerViewer(Vertical)`: bordered container for browsing topics, entries, and flashcards. Composes `TopicTree`, `EntryList`, and `FlashcardList` inside labeled panes (titles shown in multi-pane modes). Four view modes cycled by Tab: topics only, topics & entries (30/70), topics & flashcards (30/70), topics + entries + flashcards (20/35/45). Ctrl+Left/Right navigates between visible panes; Ctrl+Enter selects a topic. In 3-pane mode, EntryList uses compact mode and selecting an entry loads its flashcards in the right panel. Cursor persistence: entry cursor shared across all entry-showing modes per topic; flashcard cursor persisted per topic in two-pane mode only; three-pane flashcard cursor resets on each entry selection. `invalidate_and_refresh()` clears all data caches, reloads the topic tree, and re-fetches data for the current topic — called by `RhizomeApp.on_data_changed()` when a DB commit is detected. Posts `TopicSelected(topic, path)` and `Dismissed()`. Mounted by the `/explore` command.
- **agent_message_harness.py** — `AgentMessageHarness(Widget)`: encapsulates one agent turn's display lifecycle. Supports arbitrary interleaving of `ChatMessage`, `ToolCallList`, and `InterruptChoices` segments via `_segments`. A new segment is created at each text↔tool boundary: `append(token)` starts a new `ChatMessage` if the last segment is a `ToolCallList` (or empty), and `post_update(chunk)` starts a new `ToolCallList` if the last segment is a `ChatMessage` (or empty), closing the active `MarkdownStream` in the process. Exposes callback methods (`on_message`, `on_update`, `on_interrupt`) matching the signatures expected by `AgentSession.stream()`. `on_interrupt` mounts an `InterruptChoices` widget and blocks (via `asyncio.Future`) until the user responds; it posts `InterruptPending`/`InterruptResolved` Textual messages so `ChatPane` can toggle input. Methods: `start_thinking()`, `stop_thinking()`, `append(token)`, `post_update(chunk)`, `finalize() -> str`, `cancel() -> str`. Properties: `chat_message_body` (joined text from all `ChatMessage` segments), `is_thinking`. On finalization, calls `update_body()` on each `ChatMessage` segment to trigger collapsibility checks. Mounted synchronously by `ChatPane._start_agent()` before the agent worker starts; the worker (`_run_agent`) then drives the harness through its lifecycle. No background or border styling of its own — mode-specific borders are applied at the individual `ChatMessage` level.
- **interrupt.py** — `InterruptWidgetBase(ABC)`: base class for future-based interrupt widgets. Provides `_future`, `resolve()`, `wait_for_selection()`, `cancel()`, and `NavigableWidgetMixin` lifecycle. All interrupt widgets subclass this.
- **choices.py** — `Choices(Widget)`: displays an interrupt prompt with a navigable list of options, backed by an `asyncio.Future`. Mounted by `AgentMessageHarness.on_interrupt()` for `"choices"` type interrupts. `wait_for_selection()` awaits the future; selection resolves it. `cancel()` cancels the future for Ctrl+C cleanup.
- **multiple_choices.py** — `MultipleChoices(Widget)`: multi-question interrupt widget with tabbed navigation. Presents multiple questions as a horizontal tab bar with checkbox indicators (`[x]`/`[ ]`). Ctrl+Left/Right navigates between questions; Up/Down/Enter selects options within each question. Answering a question marks its checkbox and auto-advances to the next unanswered one. When all questions are answered, Ctrl+Enter shows a "Submit answers?" confirmation (Yes/No). Resolves the future with a `dict[str, str]` mapping question names to selected options. Mounted for `"multiple_choice"` type interrupts.
- **sql_confirmation.py** — `SqlConfirmation(Widget)`: interrupt widget for confirming SQL modification statements. Displays a warning header with row count, the SQL statement in a styled block, a preview table of affected rows (for UPDATE/DELETE), and Approve/Deny options. Follows the same pattern as `WarningChoices`: asyncio.Future for blocking, `from_interrupt` classmethod, Up/Down/Enter navigation, collapse on selection. Mounted for `"sql_confirmation"` type interrupts.
- **warning.py** — `WarningChoices(Widget)`: warning interrupt for irreversible agent actions. Displays a warning icon (⚠) and highlighted message in amber, with Approve/Deny as default options plus any additional options from the interrupt config. After selection, the warning text is hidden and only the chosen option remains visible. Mounted for `"warning"` type interrupts.
- **commit_proposal.py** — `CommitProposal(Widget)`: specialized interrupt widget for reviewing commit proposals. Master-detail layout with an entry list (navigable), detail panel (editable title/content), and choice list (Approve All/Edit/Cancel). Supports inline editing (Enter to edit, Escape to return), entry exclusion toggle (`d`), type cycling (`f`), and topic selection via modal (`t`). Mounted for `"commit_proposal"` type interrupts.
- **flashcard_proposal.py** — `FlashcardProposal(Widget)`: interrupt widget for reviewing agent-proposed flashcards. Split layout: flashcard list on the left (35%), detail panel on the right (65%) with editable question/answer/testing-notes TextAreas and a read-only linked entry IDs display, choices below. `d` toggles exclusion (strikethrough), Enter enters edit mode (question, answer, testing notes), Escape returns to browse. Choices: Approve (ctrl+a), Edit (ctrl+e, opens instruction input), Reset (ctrl+r), Cancel (ctrl+c). `from_interrupt(value)` expects `{"type": "flashcard_proposal", "flashcards": [{"question": ..., "answer": ..., "testing_notes": ..., "entry_ids": [...]}, ...]}`. Mounted for `"flashcard_proposal"` type interrupts.
- **shell_message.py** — `ShellCommandMessage(Widget)`: displays inline shell command output triggered by `!cmd` input. Shows a `you:` prefix with the command (`$ cmd`), an elapsed time indicator (visible after 10s), and a scrollable dark-background output area (max 20 lines visible) with streamed subprocess output. `execute()` runs the command via `asyncio.create_subprocess_shell` with stdout/stderr merged, stdin devnull, and a 30s timeout. Non-zero exit codes are displayed below the output area. Mounted directly into `#message-area` by `ChatPane._handle_shell_command()`.
- **tool_call_list.py** — `ToolCallList(Widget)`: collapsible list of tool call names with Unicode box-drawing characters (`├──` / `└──`). No background or border — uses a `Static` title widget ("tool calls ▼") instead of `border_title`, with left padding aligned to agent message text. Mounted lazily by `AgentMessageHarness.post_update()` when the agent invokes tools. `add_tool(name)` appends a name and re-renders the full list. Click or Enter toggles collapse: expanded shows the tree; collapsed hides content and shows "N tool calls (click to expand...) ▶".
- **chat_pane.py (commit mode)** — Commit selection is handled inline by `ChatPane` via a `_commit_mode` flag rather than a separate widget. `enter_commit_mode()` scans for `ChatMessage.agent-message.learn-mode` widgets, decorates them with checkboxes and CSS classes (`--commit-selectable`, `--commit-cursor`, `--commit-selected`), and disables the chat input. `on_key()` intercepts Up/Down/Space/Ctrl+J/Escape during commit mode. `exit_commit_mode()` cleans up decorations and restores normal input. Invoked by the `/commit` command.
- **logging_pane.py** — `LoggingPane(Widget)`: displays a rolling window of log messages captured by `TUILogHandler`. Composes a `RichLog(max_lines=2000, markup=True, wrap=True, auto_scroll=True)` and a `LogsStatusBar` hint bar docked at the bottom showing "ctrl+g to open in editor". Pull-based: on mount replays all existing lines from the handler's deque, then polls `handler.total_count` every 100ms to sync new lines. Handles deque wraparound by doing a full replay when lines have been dropped. Used inside `LogTabPane` in the chat screen, opened via the `/logs` command.
- **command_palette.py** — `CommandPalette(Widget)`: autocomplete dropdown for slash commands. Dynamically sources its command list from the parent `ChatPane`'s `CommandRegistry` (walking the widget tree to find it). Shown when input starts with `/`, filters by prefix, supports arrow-key navigation and Tab/Enter selection. Posts `CommandSelected(name)` when a command is chosen.
